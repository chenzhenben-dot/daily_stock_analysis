# 股票/证券研究框架

## Goal

对一个具体证券做一次完整的"基本面 → 估值 → 行动"分析，最后给出明确动作标签（买/加仓/持有/减仓/卖/回避/观察）和监控清单。不只是讲故事——必须回答"现在价格已经反映了多少好消息"。

## When to use

Trigger when user:
- 提到一个具体股票代码或公司名 + 询问投资判断（"NVDA 能不能买"、"分析一下 TSLA"、"云厂商哪家值得看"）
- 直接说"调研这家公司"、"research this ticker"、"fundamental analysis"
- 问"这只股贵不贵"、"现在能进吗"、"加仓还是减仓"
- 要做 portfolio 层面的对标比较

Do NOT use for:
- 加密货币分析（框架不适用，需要专门 skill）
- 纯技术面短线交易（框架强调"技术面只用于择时，不用于证明公司好"）
- 宏观/大类资产配置（需要专门框架）
- 已经被分析过的、只想要 incremental update（用 [[equity-update]] 增量更新流程）

## 数据源优先级（按可信度从高到低）

1. **公司一手**：IR 官网、10-K / 10-Q / 8-K、年报、投资者演示、earnings call transcript
2. **监管披露**：SEC EDGAR、HKEX 披露易
3. **市场数据**：Nasdaq、Yahoo Finance、StockAnalysis、Koyfin、TIKR
4. **财经媒体**：Reuters、Bloomberg、WSJ、Barron's、MarketWatch、IBD
5. **行业数据**：SemiAnalysis、Gartner、IDC、TrendForce、Synergy Research（按行业选）
6. **社区/研报**：Seeking Alpha、研报、雪球（最低优先级，需交叉验证）

> **硬规则**：单源结论必须标注"待交叉验证"。任何数字至少 2 个独立来源。

---

## 核心原则（贯穿全流程）

1. **不只说公司做什么，要说客户为什么付钱** —— 省钱？提速？扩产？降风险？合规？增收？降 TCO？
2. **不默认套 AI 产业链** —— 必须先判断公司真实在哪个产业链（光通信、半导体、云计算、航天、金融、医疗、零售消费、能源工业……）
3. **不写"某某等"** —— 产品/客户要列全，分当前收入产品 / 增长产品 / 未来期权
4. **技术面只用于择时，不用于证明公司好** —— 公司好坏看基本面
5. **价格已经反映了多少好消息** —— 估值和催化剂分开看，问的是"现在的 price 隐含了什么"
6. **数据冲突就观察** —— 不强行给结论，宁可"小仓试错"也不要硬押
7. **失效条件必须提前定义** —— 买入前就要想清楚"什么事实出现说明我错了"

## Workflow

### Phase 1：定问题

在研究开始前先问清楚 4 件事（如果用户没说，用 AskUserQuestion 问 1 次）：

1. **动作意图**：买入 / 加仓 / 持有 / 减仓 / 卖出 / 观察？
2. **持有周期**：短线（< 3 月）/ 中线（6-12 月）/ 长线（3-5 年）？
3. **证券类型**：普通股票 / ETF / 封闭基金 / SPAC / ADR / 特殊载体（DXYZ 类）？
   - **非普通经营公司**先分析结构风险：NAV、溢价/折价、流动性、持仓透明度
4. **核心问题**：用户最想知道的 1 句话是什么？（例："NVDA 的 Blackwell 出货能否支撑 2026 增长"、"这只 SaaS 股的回撤是不是机会"）

### Phase 2：研究层（把公司看懂）

按下面 10 个子模块依次填。**每个子模块必须用通俗话解释，不能只给表格/数字**。

#### 2.1 一句话定义公司

用一句中文说清：
- 这家公司卖什么？
- 客户是谁？
- 行业归属？
- 投资人真正买的增长逻辑是什么？

#### 2.2 收入结构和核心业务（**必填，看板核心 section**）

**业务结构表**（按公司分部，不是按产品族）：

| 业务/分部 | Q[N] 收入 | 占比 | 同比 | 环比 | 投资含义 |

- 必须有"占比"和"环比"——占比看结构，环比看趋势
- 同比看大趋势，环比看最近 1 季度的加速度
- 缺一个就标注"未披露"

然后**每个业务用一句话通俗类比**（参考 LITE 例子：Components 像"发动机和核心零件"，Systems 像"能直接装进网络里的成品/半成品"）：

```
Components 像"发动机和核心零件"...
Systems 像"能直接装进网络里的成品/半成品"...
```

最后给**业务结构总评**：
- 哪个业务是增长引擎（高增长 + 高占比）
- 哪个是稳定现金牛（高占比 + 低增长）
- 哪个是周期业务（中等占比 + 周期波动）
- 哪个是未来期权（低占比 + 高增长）
- 哪个是拖油瓶（高占比 + 负增长，要看是否要剥离）

#### 2.3 产品/服务模块（**必填，看板核心 section**）

产品和业务分开讲——业务是分部，产品是分部下面的具体产品族。

**产品/客户为什么付钱表**（按产品族，不是按分部）：

| 产品 | 是什么 | 客户为什么付钱 | 成功看什么 |

字段说明：
- **是什么**：用 1 句话讲清楚这个产品是什么（不超 20 字，能让非技术人听懂）
- **客户为什么付钱**：从客户视角讲解决了什么问题（提速/降本/扩产/降风险/合规/增收/降 TCO）
- **成功看什么**：可量化的指标（收入、毛利率、客户认证、量产时间、设计导入...）

产品族要**列全**，不写"某某等"。典型分法：
- **核心收入产品**：当前已经在贡献大部分收入
- **增长产品**：订单/客户/交付正在放量
- **未来期权**：市场关注但收入未充分验证
- **收尾产品**：被替代/需求下行，但还有存量收入

**整体类比框**（必填，最后一段）：

> 如果 [公司业务] 是一座 [城市/工厂/医院/...],那么这家公司的产品就是 [类比 1]、[类比 2]、[类比 3]...

类比要：
- 客户视角（不是公司内部视角）
- 体现产品之间的关联（不是孤立罗列）
- 一句话能让人记住公司是干嘛的

例子（LITE 风格）：
> 如果 AI 数据中心是一座超级城市,GPU 是工厂,交换机是交通枢纽,那么 LITE 的产品就是高速公路、光源发动机、立交桥和交通调度系统。

每个重要产品还要回答：
- 谁使用，谁付钱？
- 客户在真实工作流里怎么用？
- 公司怎么赚钱？（软件利润/制造利润/服务利润/金融利差/合约收入/周期硬件利润）
- 客户可以用什么替代方案？

**技术产品必须加通俗例子或类比。**

#### 2.4 产品完整性

不写"某某等"。按三档列全：
- **当前收入产品**：已经贡献收入
- **增长产品**：订单/客户/交付正在放量
- **未来期权**：市场关注但收入未验证

#### 2.5 产业链位置

**先判断真实产业链，不要默认套 AI**。

回答：
- 上游/中游/下游哪一段？
- 依赖哪些上游供应？
- 下游客户拿产品做什么？
- 赚的是什么钱？（软件/制造/服务/利差/合约/周期硬件）
- 风险：被客户自研？双供应商？替代技术挤压？

**确实和 AI 有关时**单独加 AI 子产业链定位：算力/互连/数据中心基础设施/云平台/模型层/应用层/服务层。

**看板上必须用多行表格形式**（反单行 3 box）—— 多元业务公司（Corning、Lumentum、Coherent 等）有多条独立产业链条，每条要单独一行：

| 上游（采购） | → | 中游（公司业务） | → | 下游（客户） |
|---|---|---|---|---|
| 石英砂/化学品 | → | Corning 光纤/光缆/连接器 | → | Hyperscaler / 电信运营商 |
| 多晶硅原料 | → | Corning (Hemlock) 多晶硅 | → | 太阳能模组厂 |
| 玻璃基板原料 | → | Corning 玻璃基板 | → | Samsung Display / LG / 京东方 |
| 聚合物 | → | Corning Gorilla Glass | → | Apple / Samsung / 汽车厂 |

不要把 5 个业务塞进一个"中游"box（局促难看）。多业务公司按"业务"分行，单业务公司按"产品族"分行。每行可以短,不需要堆数据。

#### 2.6 客户和客户验证

尽量找全：
- 官方披露的现有客户
- 历史大客户
- 未点名的 major customer / hyperscaler
- 10-K 里的客户集中度
- purchase order、design win、合作协议
- 媒体/分析师推断的客户（必须标"推断"）

重点判断：
- 是否有真实付费？
- 是否复购？
- 是否从样品 → 量产？
- 是否过度依赖单一客户？
- 是 sole-source 还是 dual-source？

#### 2.7 竞对分析

**先分四类**：
1. 直接竞争对手（卖类似产品给同类客户）
2. 间接竞争对手（解决相邻问题，争夺同一预算）
3. 替代方案（不同技术路径）
4. 客户自研/垂直整合风险

**再做三层比较**：

**Layer 1 - 财务/估值对比**：

| 公司 | 相关业务 | 收入规模 | 增速 | 毛利率 | FCF | EV/Sales | 估值备注 |

**Layer 2 - 产业链能力对比**：
- 技术代际、成熟度、订单/客户验证、产能、可靠性、成本位置、供应链控制、监管/地缘风险

**Layer 3 - 产品差异通俗解释**：
- 产品到底哪里不同？谁性能更强？谁成本更低？谁更成熟？谁客户验证更多？谁更容易被替代？
- 如果客户优先考虑价格/可靠性/性能/交付速度，谁更占优？

#### 2.8 最新财务和经营数据

至少看：
- 收入、分部收入、收入占比
- 同比/环比增长
- 毛利率、经营利润率、净利润
- EPS
- 经营现金流、capex、free cash flow
- 现金、债务、净现金/净负债
- 股本稀释
- 应收账款、库存
- 指引（guidance）
- 行业指标：backlog / RPO / ARR / bookings

**专业术语先解释**（RPO = Remaining Performance Obligation，未来合同义务剩余额；ARR = Annual Recurring Revenue，年度经常性收入；bookings = 已签约订单）

#### 2.9 估值和市场定价

- 市值、企业价值（EV）
- P/E、Forward P/E
- EV/Sales、Forward EV/Sales
- EV/EBITDA
- FCF yield
- 同业估值
- **当前价格已经隐含了什么增长和利润率**

**不能只说"故事好"，要问：现在 price 反映了多少好消息？**

#### 2.10 催化剂 vs 基本面

**分四类列**：

| 类别 | 例子 | 影响维度 |
|---|---|---|
| 经营催化剂 | 财报超预期、指引上调、新客户、产品放量 | 收入/利润/现金流 |
| 财务催化剂 | 回购、分红、融资、债务再融资、指数纳入 | 估值/资本结构 |
| 叙事催化剂 | 政治人物提到、社交媒体热度、分析师上调、板块轮动 | 短期情绪 |
| 结构催化剂 | 政策、预算周期、供应链变化、行业技术迁移 | 中长期趋势 |

**关键判断**：每个催化剂是改变收入/利润/现金流，还是只改变短期情绪？

### Phase 3：动作层（判断能不能买）

#### 3.1 周期位置

判断公司在哪个阶段：

- **早期反转**：基本面还弱，但库存、价格、订单、供给开始改善
- **中期确认**：收入、订单、毛利、指引同步改善，股价趋势也确认
- **后期亢奋**：财报很好、故事很热、估值扩张、市场预期很高
- **高景气尾段/周期顶部**：财报还好看，但领先指标变弱，供给追上需求，利润率不再改善
- **不清楚**：数据冲突，只能观察或小仓试错

#### 3.2 买入前事实清单（必须逐条核对）

| 维度 | 关键问题 |
|---|---|
| 行业周期 | 早期 / 中期 / 后期 / 顶部？ |
| 供给端 | 产能、capex、新产能、库存、供应链约束 |
| 需求端 | 订单、出货、价格、客户结构 |
| 公司端 | 收入、毛利、现金流、债务、库存、应收、稀释、指引 |
| 竞争端 | 产品代际、客户认证、价格战、客户自研 |
| 市场端 | 板块强弱、相对大盘、成交量、空头比例、资金流 |
| 估值端 | 是否已经提前透支？price 隐含了什么？ |
| **失效条件** | 什么事实出现说明买入逻辑错了？ |

#### 3.3 卖出/减仓框架（提前定义，不是等跌了才想）

触发的常见信号：
- 订单消失 / 收入低于预期 / 毛利率恶化
- backlog 质量下降 / 大客户流失
- 指引下调 / 库存+应收恶化
- FCF 持续变差 / 债务或稀释压力上升
- 竞品领先 / 价格战 / 估值过度透支
- 情绪过热 / 技术面放量破位

#### 3.4 技术面和资金面（只用于择时）

- 趋势、相对强弱、成交量
- 支撑/阻力、短期涨幅
- 板块资金流、short interest、是否过度拥挤

## Phase 4：动作标签（必须给一个）

从下面 8 个选 1 个，并简述理由：

| 标签 | 触发条件 |
|---|---|
| **可以买/加仓** | 基本面改善，周期早中期，估值还能接受，失效条件清晰 |
| **小仓试错** | 早期信号有，但还没完全验证 |
| **等待回调** | 公司逻辑成立，但价格/情绪/估值太满 |
| **持有** | 逻辑还在，但新买不划算 |
| **减仓** | 后期、估值高、情绪拥挤，风险回报变差 |
| **卖出** | 逻辑破坏、周期反转、财务压力或竞争恶化 |
| **回避** | 只有故事，没有数据证据 |
| **观察名单** | 公司值得看，但时机或数据还不够 |

---

## Phase 5：监控清单（每次结尾必给）

1. **下一次财报日期或预计窗口**
2. **当前季度/全年指引**
3. **下次财报最该看的 3-6 个指标**（按重要性排序）
4. **产品交付/客户订单节点**（design win、量产、capex 进度等）
5. **估值需要验证的关键假设**（如"如果 Blackwell 2026 出货 100 万卡，EV/Sales 降到 X 才合理"）
6. **买入逻辑失效条件**（Phase 3.2 的最后一条展开）
7. **卖出/减仓条件**（Phase 3.3 的信号清单展开成可观测指标）

## 形式 A：HTML 看板（默认）

### 模板位置

`references/dashboard-template.html` —— 完整的 HTML 模板（CSS/JS 内嵌，Bloomberg 深色风，6 个 KPI 卡 + 7 个 tab）。

### 填充规则

1. Read `references/dashboard-template.html` 拿模板
2. 替换所有 `{{placeholder}}`（包括 `{{action_class}}`、`{{ytd_class}}` 等条件 class 名）
3. 动态生成 `<tr><td>...</td></tr>` 行时，按已有 `<tr>` 行的格式拼
4. 写入本地文件（见下方"输出位置"）
5. 对话里给一段简短总结 + 文件路径

**生成器**：`references/dashboard-generator.py` 接受 JSON 数据 + 模板路径，自动写文件。
用法：`python3 dashboard-generator.py <data.json>`

### 输出位置

```
~/Desktop/equity-dashboards/<TICKER>-<YYYY-MM-DD>.html
```

- 目录不存在就创建
- 同一天同一只股二次分析 → 覆盖（用当天日期）
- 跨天同股 → 保留多份（用户做时间序列对比）

### 看板结构（7 个 tab）

1. **概览** — Header + KPI 卡 + 一句话结论 + 公司定义 + 投资逻辑 + 关键数据快照 + 价格隐含
2. **财务估值** — 最新季报 + 全年 + 分业务收入 + 指引 + 估值 + 资产负债表
3. **产业链** — 归属 + 上下游图 + 赚钱模式 + 4 类风险
4. **竞对** — 4 类分法 + 财务对比 + 通俗差异 + 客户优先看 X 谁占优
5. **客户验证** — long-term 协议表 + 5 维验证（付费/复购/量产/集中度/sole-dual）+ 风险
6. **周期催化** — 业务周期 vs 股价周期 + 4 类催化剂 + 关键判断

### 数值颜色规则

- 正向变化（增长、超预期、看好）→ class="pos"（绿）
- 负向变化（下滑、未达预期、看空）→ class="neg"（红）
- 警示（透支、过高、待观察）→ class="warn"（黄）
- 中性信息 → 不加颜色 class

### 文本内容规则（**保持 markdown 报告同样的内容纪律**）

- 不写"某某等"
- 客户/产品/业务要列全
- 不只给数字不解释
- 失效条件必须定义
- 单源结论标注"待交叉验证"
- 监控清单不能省
- **催化剂事件要写完整句子**（**反短句**）—— 事件列不能只写"NV $2B 投资"或"record revenue +90% YoY"。必须包含**具体数据 + 时间 + 含义/影响**。例如：
  - ❌ "NVIDIA $2B 投资"
  - ✅ "2026-03-02 NVIDIA 以 Series A 可转优先股向 LITE 投资 $20 亿美元,同期签多年 non-exclusive 协议(明示 NVIDIA 'multi-billion' 采购承诺 + 优先获得先进激光器产能 + R&D 合作)。Lumentum 拿钱在美国本土建新 fab"
  - 13 条催化剂每条都要展开成 1-2 句,合计 200-400 字的"事件 + 数据 + 含义"叙述

1. 文件路径（让用户点开浏览器）

## Hard rules（违反任何一条 = 重做）

1. **不写"某某等"** —— 客户、产品、业务都要列全
2. **不默认套 AI 产业链** —— 先判断真实归属
3. **不只讲故事不给数字** —— 收入、毛利、FCF、估值必须有
4. **不只给数字不解释** —— 数字后必须配通俗话讲"为什么"
5. **不忽略失效条件** —— 买入前必须定义
6. **不把技术面当基本面用** —— 技术面只用于择时
7. **数据冲突时不强行给结论** —— 给"小仓试错"或"观察名单"
8. **单源结论必须标注"待交叉验证"**
9. **不混用周期阶段** —— 早期/中期/后期/顶部有清晰定义，不能模棱两可
10. **监控清单不能省** —— 每次结尾必给
11. **数据必须可溯源，估算必须标注**（**反臆造**）—— 任何数字必须是公司披露/媒体/SEC 公开数据。**找不到的数据写"未单独披露"**，不允许凭感觉写。占比用同口径基数（GAAP 或 Core）算，反推数据要标"反推"。**这是投资决策依据，臆造数字会直接误导交易**
12. **类比/比喻不算数据** —— 业务类比（血管/地基/防弹衣）和整体类比（超级城市）是便于理解的比喻，**不是客观事实**。"业务 & 产品" tab 顶部必须加 disclaimer："以下类比便于理解,非客观数据"
13. **占比和环比必须可算** —— 占比用公司披露的同口径基数算（不能凭印象）；环比必须有连续 2 个季度的同口径数据才能写，否则写"未披露"
14. **看板生成前必须做字段完整性检查**（**反漏字段**）—— 给新公司生成看板前，对照模板里的所有 `{{placeholder}}` 字段，确认 JSON 里都有对应数据。常见必填字段：`chain_overview` + `chain_rows`（产业链 tab）/`business_quarter_label` + `business_rows` + `business_analogies` + `business_summary`（业务 & 产品 tab）/`product_rows` + `product_analogy_box`（产品子 section）。**漏一个字段 = 模板渲染时该 section 只剩表头**

## Notes

- 用户中文沟通为主，报告主体中文写，**英文专业术语保留**（EV/EBITDA、backlog、design win 等不翻译）

- 给"可以买/加仓"标签时**必须**输出 V3 风格的"买入前 8 问 + 单页下单检查表 10 条"，让用户自己走一遍纪律流程
